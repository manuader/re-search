# ResearchBot — Product Requirements Document

## 1. Product Vision

ResearchBot is a web platform where anyone describes in natural language what they want to research, and an intelligent chatbot guides them to extract data from the internet, analyze it with AI, and generate actionable reports. All with transparent costs and a pay-per-use model.

**Positioning: "Deep Research as a Service"** — not a social listening tool, not a scraping platform. It's a conversational intelligence layer that transforms a research need into structured data and reports, without technical knowledge.

### User Flow

```
1. Login → New Project → Chat with bot → "I want to research X"
2. Bot interprets the need and suggests tools from curated catalog
3. User selects tools → sees estimated cost in real time
4. Configures keywords, filters, volume (guided by the bot)
5. Optionally adds AI analysis (sentiment, classification, etc.)
6. Sees final cost summary → Accepts
7. Extraction runs → real-time progress
8. Receives Excel/CSV with raw data + AI fields
9. Generates interactive HTML report with insights
```

### Target Users

- Marketing/PR consultants (one-off research for clients)
- Academic researchers and data journalists
- Startups validating ideas
- Small agencies that don't pay $800/month for Brandwatch
- Government teams analyzing public perception
- Product managers researching feedback

## 2. Technical Decisions

| Decision | Choice | Rationale |
|---|---|---|
| Auth | Google OAuth + email/password fallback | Supabase Auth handles everything, Google covers most target users |
| Payments | Mercado Pago first, Stripe later | LATAM market first, MP has local payment methods |
| MVP Tools | 13 from curated catalog | Good coverage across maps, social, search, ecommerce, travel, professional |
| Language | English default, i18n from day 1 | next-intl, prefix-based routing, easy to add languages |
| Realtime | Supabase Realtime (postgres_changes) | Already in stack, zero additional infra |
| Reports | Interactive HTML generated by 1 LLM prompt | Single prompt to Sonnet → complete HTML with Recharts via CDN |
| PDF Export | None for MVP | User can print-to-PDF from browser |
| Credits | Balance = SUM(transactions) | No mutable balance field, transactions are source of truth |
| Chat | One chat per project | Bounded context, simpler history management |
| LLM — Chatbot | Claude Haiku 4.5 (via @ai-sdk/anthropic) | Only selects tools + configures params. Single API key for all LLM usage. ~$0.005/conversation |
| LLM — Reports | Claude Sonnet 4.6 (via @ai-sdk/anthropic) | Report quality matters — needs strong reasoning for insights and HTML generation |
| LLM — Analysis | Claude Haiku 4.5 (Batch API) | Cheap, accurate in Spanish, 50% off with Batch API |
| Tool Calling UX | Semi-transparent indicators | Shows "Searching tools..." etc. while tools run |
| AI Analysis | Anthropic Batch API always | 50% cheaper, flow is already async |
| Tool Catalog | Hybrid: static code + dynamic DB health | Definitions in TypeScript, health status in DB |
| Architecture | Next.js monolith + Inngest for background jobs | Single deploy, Inngest handles serverless timeouts |
| Billing Rule | Never refund more than what APIs cost us | Discounts reduce our margin, never generate loss |

### Data Flow

1. User talks to chatbot → API route `/api/chat` → Claude Haiku 4.5 with tool calling
2. Claude calls tools (searchTools, estimateCost, etc.) → resolves against static catalog + DB health
3. User confirms → API route fires Inngest event
4. Inngest runs scraping (Apify API) → polling → saves data to Supabase
5. Supabase Realtime notifies frontend of progress
6. Post-scraping: Inngest fires batch analysis (Haiku 4.5 Batch API)
7. User requests report → single prompt to Sonnet → interactive HTML
8. Export → generates Excel/CSV → uploads to Supabase Storage

## 6. Chatbot Architecture

**Model: Claude Haiku 4.5** via `@ai-sdk/anthropic`. The chatbot's only job is tool selection, parameter configuration, keyword suggestion, and cost estimation. It does NOT generate reports or analyze data — those use Claude Sonnet 4.6 and Haiku 4.5 respectively.

### System Prompt

Built per-locale from `lib/ai/system-prompt.ts`. Template:

```
You are the ResearchBot assistant, a platform for researching public data on the internet.

Your job:
1. Understand what the user wants to research
2. Suggest the best tools from your catalog
3. Configure parameters conversationally
4. Show estimated costs transparently
5. Suggest AI analyses that add value

Rules:
- Never mention "Apify", "Actor", or technical scraping terms
- Call tools by their public name ("Business Finder")
- Always show estimated cost before executing
- If a tool is "degraded", warn the user
- If a tool is "down", suggest alternatives
- For technical fields (proxy, headers, cookies), configure silently
- Suggest keywords in the user's language including colloquial variations
- Maximum 1-2 questions per message to avoid overwhelming
- Respond in {locale} language
```

### Tool Definitions (6 tools)

**1. searchTools(query: string)**
- Fuzzy search against static catalog array
- Returns matching tools with health status from DB
- Used when user describes what they want to research

**2. getToolConfig(toolId: string)**
- Returns input schema for a specific tool from static catalog
- Claude decides which fields to ask user about (userFacing: true) vs auto-configure

**3. estimateCost(toolId: string, config: object)**
- Pure function: pricing model × requested volume
- Returns { min, max, expected } cost range

**4. suggestKeywords(objective: string, toolId: string)**
- Claude generates optimal keywords including semantic variations, slang, hashtags, misspellings
- Language-aware (handles lunfardo, modismos, local expressions)

**5. suggestAIAnalysis(objective: string, dataType: string)**
- Suggests relevant AI analysis templates (sentiment, classification, entities, etc.)
- User can accept, modify, or add custom analysis fields

**6. executeResearch(projectId: string)**
- Triggered after user confirms
- Verifies credit balance ≥ estimated cost
- Reserves credits (INSERT transaction: scraping_reserve)
- Creates scraping_jobs records
- Dispatches Inngest event: "research/execute"
- Returns confirmation with estimated time

### Semi-Transparent Tool Call UX

While tools execute, the UI shows status indicators:
- "🔍 Searching tools..." (during searchTools)
- "⚙️ Loading configuration..." (during getToolConfig)
- "💰 Calculating costs..." (during estimateCost)

After the tool completes, the indicator collapses to a compact "✓ Searched tools catalog" and the assistant's full message appears below.

`tool_invocations` JSONB in `chat_messages` stores tool call data for UI rehydration on page reload.

## 7. Execution Pipeline (Inngest)

### Function 1: `research/execute` — Main Pipeline

```
Trigger: Inngest event "research/execute" { projectId }

Step 1: Reserve credits
  → INSERT transaction (type: scraping_reserve, amount: -estimated_total)
  → UPDATE research_projects.status = 'running'

Step 2: For each scraping_job (sequential):
  → 2a: Start Apify actor run (POST /v2/acts/{actorId}/runs)
         Save apify_run_id
         UPDATE scraping_jobs.status = 'running'

  → 2b: Poll until done (step.sleep 30s, max 60 attempts = 30 min)
         Each poll updates scraping_jobs → Supabase Realtime → UI

  → 2c: Download dataset (GET /v2/datasets/{id}/items)

  → 2d: Validate data quality
         Remove duplicates, check required fields, verify integrity
         Set quality_score: high | medium | low

  → 2e: Batch insert into raw_data (chunks of 100)

  → 2f: Update scraping_jobs (actual_results, actual_cost, status)

Step 3: Reconcile costs
  → Calculate total actual cost (all jobs)
  → Apply markup: user_cost = actual_cost × 1.4
  → INSERT transaction adjustment (difference between reserved and user_cost)
  → INVARIANT: user always pays ≥ actual API cost

Step 4: Run AI analysis (if configured)
  → Build prompts for all raw_data rows
  → Submit to Anthropic Batch API
  → Save batch_id to ai_analysis_configs
  → Dispatch "research/check-batch" event

Step 5: If no AI analysis → mark project completed
```

### Function 2: `research/check-batch` — Batch API Polling

```
Trigger: Inngest event "research/check-batch" { batchId, projectId }

Step 1: Check batch status (GET /v1/messages/batches/{batchId})
  → If processing: sleep 60s, retry (max 24h)
  → If ended: download results

Step 2: Parse results → update raw_data.ai_fields

Step 3: Update ai_analysis_configs (status, actual_cost)

Step 4: If all configs complete:
  → INSERT transaction (type: ai_cost)
  → UPDATE research_projects.status = 'completed'
```

### Function 3: `cron/health-check` — Automated Monitoring

```
Trigger: Inngest cron "0 */12 * * *" (every 12 hours)

For each tool in static catalog (sequential):
  → Run mini-test with health_check_input
  → Record in actor_health_log
  → Update actor_health:
    - success → reset consecutive_failures, recalc rates
    - failure → increment consecutive_failures
    - 3 consecutive failures → status = 'down', is_available = false
    - success_rate_7d < 80% → status = 'degraded'
    - success_rate_7d > 90% → status = 'healthy'
```

### Error Handling

- If a job fails, other jobs continue. Project completes with some jobs in "failed" status.
- User is charged only for what was consumed (actual API cost × 1.4).
- Each Inngest step is independently retriable. If dataset download fails, the actor doesn't re-run.
- 30-minute timeout per actor. If exceeded, job marked as failed.

## 8. Credits & Payments

### Purchase Flow

1. User goes to /billing → sees balance (SUM of transactions)
2. Selects amount: $5 (min), $10, $25, $50, or custom
3. Click "Buy" → creates Mercado Pago preference (Checkout Pro)
4. Redirect to MP hosted checkout
5. User pays → MP webhook to /api/webhooks/mercadopago
6. Webhook verifies HMAC signature → INSERT transaction (credit_purchase, +amount)
7. Redirect back to /billing

### Reservation & Billing Logic

```
On research confirm:
  1. Verify balance ≥ estimated cost
  2. If insufficient → show message + link to /billing
  3. If sufficient → INSERT transaction (scraping_reserve, -estimated)

On research complete:
  4. actual_api_cost = sum(actual_cost of all completed jobs + AI)
  5. user_cost = actual_api_cost × 1.4 (40% markup)
  6. adjustment = estimated - user_cost
  7. INSERT transaction (scraping_cost, +adjustment)
     → If user_cost < estimated: positive adjustment (refund difference)
     → If user_cost > estimated: negative adjustment (charge extra)

On research failure (before consuming APIs):
  8. Full refund: INSERT transaction (refund, +estimated)

On research failure (after consuming APIs):
  9. Charge consumed: user pays actual_api_cost × 1.4
  10. Refund only the unconsumed portion

On partial results (<50% expected):
  11. Proportional charge with reduced markup:
      user_cost = actual_api_cost × 1.32 (markup reduced from 40% to 32%)
  12. INVARIANT: user_cost ≥ actual_api_cost (discount only reduces our margin)
```

### INVARIANT: Never Lose Money

`charge_to_user ≥ actual_api_cost` — ALWAYS. Discounts for poor quality only reduce our markup percentage, never the base cost. If APIs charged us $20, the user pays at minimum $20.

### Free Tier

On profile creation (Supabase trigger):
→ INSERT transaction (credit_purchase, +$3.00, 'Welcome credits')

$3 allows one small test research. Regular credits, no special logic.

## 10. Internationalization (i18n)

### Setup: next-intl with prefix-based routing

```
researchbot.com/en/dashboard    → English
researchbot.com/es/dashboard    → Spanish
researchbot.com/                → redirect to /en (or user's locale)
```

### File Structure

```
src/
├── i18n/
│   ├── request.ts              -- next-intl server config
│   └── routing.ts              -- locale routing config
├── messages/
│   ├── en.json                 -- English (default)
│   └── es.json                 -- Spanish
```

### What Gets Translated

| Layer | Translated? | How |
|---|---|---|
| UI (buttons, labels, headers) | Yes | messages/{locale}.json + useTranslations() |
| Chatbot system prompt | Yes | Template per locale in lib/ai/system-prompt.ts |
| Chatbot responses | Automatic | Claude responds in the system prompt's language |
| Tool public names | Yes | name field in catalog is Record<Locale, string> |
| Transactional emails | Yes | Templates per locale |
| Reports HTML | Automatic | Prompt to Claude includes user's locale |
| Scraped data (raw_data) | No | Data is what it is |
| API error messages | Yes | Translated keys in response |

### Locale Detection

1. User's `locale` in profile (if logged in)
2. `Accept-Language` header (if not logged in)
3. Fallback: `en`

### Adding a New Language

1. Create `messages/pt.json` (copy from en.json, translate)
2. Add system prompt template for locale
3. Add translations to tool names in catalog
4. Add locale to supported locales list

## 11. Export & Reports

### Excel/CSV Export

1. API route `/api/export` receives `{ projectId, format: "xlsx" | "csv" }`
2. Query raw_data WHERE project_id (with ai_fields if present)
3. Flatten JSONB → flat columns (content fields + ai_ prefixed AI fields)
4. Generate file: xlsx via exceljs (headers bold, autofit columns, Arial 10) or CSV
5. Upload to Supabase Storage → bucket "exports"
6. Return signed URL (expires 24h)

For large datasets (>10K rows): stream in chunks to avoid serverless memory limits.

### Interactive HTML Report

1. API route `/api/report` receives `{ projectId }`
2. Prepare context: aggregate raw_data into summary JSON (~2-4K tokens):
   - Totals, distributions, top items
   - Sentiment distribution, categories (if AI fields exist)
   - Metadata: tools used, volume, date range
   - 10 representative sample items
3. Single prompt to Claude Sonnet 4.6:
   - Input: aggregated data JSON + project title + locale
   - Output: complete self-contained HTML with Recharts (CDN), React/ReactDOM (CDN), inline styles
   - Includes: executive summary, key insights, charts (Pie, Bar, Line), actionable recommendations
4. Save HTML to `reports.html_content`
5. Serve in `/projects/[id]/report` via sandboxed iframe:
   ```html
   <iframe sandbox="allow-scripts" srcdoc={html_content} />
   ```
   sandbox blocks navigation, forms, parent DOM access. Scripts allowed for Recharts.

## 12. Security

### Auth Flow

1. User lands on /login → Google OAuth or email+password
2. Supabase Auth handles the full flow
3. Next.js middleware verifies session on each request
4. No session → redirect to /login

### Middleware

- `/[locale]/(dashboard)/*` → require auth
- `/[locale]/(auth)/*` → public
- `/api/*` → verify auth header (except `/api/webhooks/*` and `/api/inngest`)

### Rate Limiting

- Chat API: max 30 messages/minute per user
- Execute API: max 5 executions/hour per user
- Export API: max 10 exports/hour per user

Implementation: Vercel native rate limiting (Pro plan) or simple counter in Supabase.

### Webhook Security

- Inngest: signing key verified automatically by SDK
- Mercado Pago: verify `x-signature` header with HMAC

## 16. Risks & Mitigations

| Risk | Probability | Mitigation |
|---|---|---|
| Apify actor stops working | Medium | Health monitoring + auto-disable + degraded warnings |
| Apify raises prices | Low | Markup absorbs variation, evaluate Crawlee own scrapers |
| Low quality data | Medium | Post-scraping validation + proportional billing |
| Chatbot misinterprets intent | Medium | Prompt engineering, iterative conversation design |
| Twitter/X blocks scraping | High | Warning to user, suggest alternatives |
| Users don't understand the system | Medium | Chatbot guides everything, semi-transparent UX |
| Batch API takes too long | Low | 24h max wait, user notified, can view partial data |
